# Coding Standards — obsidian-chat

_Generated by claude-scaffolder. Re-run `/claude-scaffolder` to refresh._

## TypeScript

- `strict` mode enabled in `tsconfig.json` — no implicit `any`, strict null checks required
- Never use `any`; use specific types or extend existing interfaces; use `unknown` only as a last resort
- All interfaces must be prefixed with `I` (e.g., `IInlineCommand`, `IChatViewState`)
- Path alias `@/*` maps to `src/*` — always use `@/components/Foo` style imports, never deep relative paths

## Naming Conventions

| Context | Convention | Example |
|---------|-----------|---------|
| React components, hooks, props, variables | `camelCase` | `useStreamChat`, `activeTabId` |
| Utility/lib files (`lib/`, `utils/`) | `snake_case` | `save_chat`, `parse_chat_note` |
| Interfaces | `IPascalCase` | `IChatTab`, `IStreamChatOptions` |
| CSS classes (plugin-specific in `styles.css`) | `oac-` prefix | `oac-message-block` |
| Tailwind utility classes in TSX | `chat:` prefix | `chat:flex`, `chat:h-full` |

## React

- All components must be render-optimized: use `useCallback`, `useMemo`, and `memo` to eliminate unnecessary re-renders
- All state is local React state in `App.tsx` — no global store
- `useRef` for abort controllers and scroll tracking (not state, to avoid re-renders)
- `ObsidianAppContext` provides the Obsidian `App` instance — consume via `useObsidianApp()` hook, never prop-drill

## CSS / Styling

- Tailwind v4 with `chat:` namespace — every Tailwind utility must be prefixed: `chat:flex`, not `flex`
- Custom plugin CSS in `src/styles.css` uses `oac-*` class names
- Never use unprefixed Tailwind classes — they will not be generated and will silently have no effect

## Build Constraints

- `obsidian`, `electron`, `@codemirror/*`, `@lezer/*` must remain external in esbuild — never import in a way that forces bundling
- `main.js` and root `styles.css` are compiled artifacts that must be committed — they are not in `.gitignore`

## General

- Follow DRY principles — no duplicate logic across files
- Minimize comments — only explain non-obvious intent or trade-offs, never narrate what the code does
- Never use placeholder/stub implementations — all code must be production-ready
- No linter or formatter is currently configured (no ESLint, Biome, or Prettier)
